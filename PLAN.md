# PLAN.md — 시스템 개선 사이클 계획

> 작성일 2026-06-12. 모의투자 검증(18/30건) 단계에서 실전 전환까지의 개선 계획.
> 기존 문서와의 관계는 다음과 같다. analysis_plan.md(30건 후 검증 항목 원본), checklist.md(구축기 기록), real_trading_transition.md(실전 전환 체크리스트)는 그대로 유지하고, 이 문서는 그 사이를 잇는 실행 계획이다.

## 목표

1. 쌓인 데이터로 전략이 설계 의도대로 작동하는지 진단한다 (파라미터 판정은 30건 후).
2. 전략 수치는 건드리지 않고 시스템 안정성·리스크 구조만 보완한다.
3. 30건 검증 때 필요한 데이터 컬럼을 지금 추가해 수집을 시작한다.
4. 30건 도달 시 데이터 기반으로 전략 변경을 검토하고 시스템에 반영한다.

## Phase A — 현재 데이터 진단 분석 (데이터 수신 후 즉시)

표본 18건으로 승률·손익비를 판정하지 않는다. 사례 진단과 필터 검증만 한다.

| 분석 | 데이터 | 보려는 것 |
|------|--------|----------|
| 이익 반납률 | trades.csv (peak_price vs exit_price) | +7%~+20% 구간 트레일 -12%가 너무 넓은지 정황 |
| 매도 사유별 결과 | trades.csv (reason) + followup_log.csv | 손절/트레일/본전보호 매도 후 가격이 실제로 더 빠졌는지 |
| 필터 효율 | signal_log.csv (탈락 종목 포함 전수) | BB %B, 외국인, 거래량 필터가 나쁜 종목을 거르는지, 좋은 종목까지 거르는지 |
| 진입 방식 비교 | timing_log.csv + trades.csv (dip_entry_used) | 눌림목 진입 vs 09:30 완화 진입 성과 차이, 10:00 포기 종목의 이후 흐름 |
| 보유기간 | trades.csv (entry_date~exit_date) | 시간손절 20일이 모멘텀 소멸 대비 너무 긴지 |

→ 검증 기준: 분석 결과를 PROGRESS.md에 기록하고, 전략 변경 후보를 "30건 후 재확인 목록"으로 TASKS.md에 적재.

## Phase B — 시스템 보완 (전략 불변, 즉시 가능)

검증 표본을 오염시키지 않는 안정성·리스크 보강만 포함한다.

1. EMERGENCY_STOP_RATE 정합화 — 서버 .env 확인 후 코드 기본값과 문서(-15%) 일치시킴
2. 매수 재시도 중복 주문 방지 — 주문 접수 성공 여부 기준으로 재시도 분기
3. positions / realized_pnl 파일 영속화 — 재시작 시 peak_price·break_even_set 유실 해결
4. 일일 손실 서킷브레이커 — 당일 손실이 총자산 일정 비율 도달 시 신규 매수 중단
5. 동일 테마 종목 수 제한 (최대 2종목)
6. 토큰 만료 여유 60초 → 300초
7. positions 접근 threading.Lock 보호

→ 검증 기준: 각 항목 수정 후 모의투자 환경에서 1거래일 무에러 확인. 항목별 시맨틱 커밋.

## Phase C — 최적화용 데이터 수집 확장 (빨리 넣을수록 가치 있음)

30건 시점 검증에 필요하지만 현재 기록되지 않는 항목을 추가한다.

1. signal_log에 테마 연속 상승일수 컬럼 (analysis_plan 9번 — 과열 추격 패턴 분리용)
2. trades에 이익 반납률(peak 대비) 파생 컬럼 또는 분석 스크립트
3. 매도 시점의 KOSPI 추세·VKOSPI 스냅샷 기록 (시장 국면별 성과 분리용)

→ 검증 기준: 다음 거래일 로그에 신규 컬럼이 정상 기록되는지 확인.

## Phase D — 30건 도달 후: 전략 변경 검토 및 반영

analysis_plan.md의 8개(+9번) 검증 항목을 그대로 실행하되, Phase A의 진단 결과와 교차 확인한다.
주요 검토 후보(현재 가설, 데이터로 확정).

- 트레일링 -12% 폭 / ATR 승수 1.5×
- 시간손절 20거래일 단축 여부
- BB %B 임계값 0.95 / 0.85
- 본전 보호 +7% → +2% 구간의 이익 반납 구조

변경 확정 시 real_trading_transition.md 체크리스트와 함께 실전 전환 판단.

→ 검증 기준: 변경 전/후 파라미터와 근거를 context-notes.md에 기록, 변경 후 새 검증 사이클 시작.
